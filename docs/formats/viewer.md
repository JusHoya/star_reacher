# Viewer HTML and the srview embedded-data schema

Normative specification of the `star view` output (PRD FR-19, decision D-16):
a **self-contained single-file HTML WebGL playback viewer**, generated by
`star_reacher.viewer.generate_view` from one SRLOG file. The embedded data
block follows the versioned `srview` schema in section 3. Playback consumes
only the log -- nothing is ever re-simulated.

## 1. Guarantees

- **Self-contained, zero network requests.** The file embeds the vendored
  three.js runtime (MIT, pinned in
  `python/star_reacher/_viewer/vendor/PROVENANCE.md`), the view stream, the
  application code, and the coastline overlay
  (`python/star_reacher/_assets/`, public domain). Textures are generated
  procedurally on a canvas at load. No `src`/`href` fetches an external URL,
  there are no `<link>`/`<img>`/`<iframe>` elements, and no network API is
  invoked: `scan_external_references` in `star_reacher.viewer` performs the
  static position-restricted scan, `star verify` runs it as part of V019,
  and `tests/python/test_viewer.py` re-asserts it with an independently
  written scan. The three.js ES module is loaded through a Blob URL built
  from an inline `<script type="text/plain">` block -- same-document memory,
  not a request -- so the vendored bytes stay verbatim against their pinned
  SHA-256.
- **Deterministic.** The HTML bytes are a pure function of the log bytes and
  the committed template/vendor/asset files: no timestamps, no filesystem
  paths, no dict-order nondeterminism. Regenerating from the same log is
  byte-identical (gated by V019 and `test_regeneration_byte_identical`).
- **Display-only interpolation.** Between keyframes the viewer interpolates
  position linearly in time and attitude by spherical-linear interpolation
  (slerp, shortest arc). This is **non-physical** -- it is not the propagated
  dynamics -- and is labeled as such in the viewer HUD and help overlay.
  Analysis must read the log, never the viewer stream.

## 2. Scrub-extreme epoch contract

The HUD epoch at scrub position 0 and at the end must equal the log header's
first/last epochs exactly (Phase 5 exit criterion 2). The generator derives
both **directly from header fields**, never by float accumulation:

- `epoch.utc_first` -- the header `epoch_utc` string **verbatim** when the
  first truth record sits at t = 0 (always true for core-written logs);
  otherwise one datetime addition of the first record time.
- `epoch.utc_last` -- one datetime addition: `epoch_utc` + the final truth
  record's `t_s`, formatted `YYYY-MM-DDTHH:MM:SS[.ffffff]Z` with trailing
  fractional zeros trimmed.

The viewer returns these strings verbatim at the scrub extremes; between
them the displayed epoch is millisecond-grade `Date` arithmetic. Leap
seconds inside the span are not applied to the display (calendar arithmetic
on the header epoch is the documented convention; the core's timekeeping is
unaffected).

## 3. Embedded data: schema `srview` version 1

The HTML carries one JSON document in
`<script type="application/json" id="srview-data">`. Every `</` inside it is
written as the JSON escape `<\/`, so `</script>` cannot occur inside the
block; standard JSON parsers read it unchanged. Fields:

| Field | Content |
|---|---|
| `schema` | `{"name": "srview", "version": 1}`. Version bumps follow SRLOG discipline: additive keys = same version, layout change = version bump. |
| `header` | `epoch_utc`, `central_body`, `config_sha256`, `master_seed`, copied from the log header (the reproducibility anchor rides along). |
| `epoch` | `utc_first`, `utc_last` (section 2), `t_first_s`, `t_last_s` (first/last truth `t_s`, bit-exact). |
| `body` | `name`; `radius_m` (display radius, section 5; `null` for an unknown body); `rotation` = `{"model": "era", "era0_rad", "rate_radps"}` for Earth, else `{"model": "none"}`. |
| `decimation` | `algorithm` (name string), `bound_m`, `measured_max_error_m`, `position_span_m`, `kept`, `total` (section 4). |
| `frames` | The keyframe stream: `t_s` (n), `r_m` (n x 3), `v_mps` (n x 3), `q_i2b` (n x 4, Hamilton scalar-first, GCRF->body). Every keyframe is a truth record of the log, bit-exact; attitude and velocity keyframes ride the position schedule. |
| `events` | `t_s`, `code`, `detail` -- the log's events stream, complete and bit-exact (the scrub-bar ticks). |
| `forces` | `null` when the log has no `forces` group. Otherwise `t_s`, `stride`, `sources` (declaration order), `f_b_n` (per source: m x 3 body-frame force [N]). Records are embedded 1:1 up to 20,000, then stride-decimated (`stride` > 1 records the step). Torque channels are not embedded. |
| `coastline` | The parsed `_assets/ne_110m_coastline.json` document for Earth runs, else `null`. |

Floats serialize with Python `repr` semantics (shortest round-trip), so
embedded values compare bit-exact against the log after `JSON.parse`.

## 4. Decimated view stream

- **Bound.** `bound_m = max(100 m, 0.01 % of the position span)`, where the
  span is the bounding-box diagonal of all truth positions
  (`|max(r) - min(r)|`). This is the Phase 5 exit-criterion-2 bound.
- **Algorithm.** Greedy largest-error keyframe insertion -- Douglas-Peucker
  on the time-parameterized 3D polyline. Starting from the two endpoints,
  the dropped sample with the largest distance from time-linear
  interpolation between its bracketing keyframes is promoted, recursively,
  until every segment's worst error is <= the bound. Quiet coasts collapse
  to a handful of keyframes; burns keep dense support. Deterministic (ties
  resolve to the lowest index).
- **Measurement.** After decimation the generator recomputes the maximum
  interpolation error over **every dropped truth sample** (exact arithmetic
  on the log values, the same interpolant playback uses) and embeds it as
  `measured_max_error_m` next to `bound_m`; `star view` prints both. The
  test suite recomputes the measurement independently from the log and the
  embedded keyframes.

## 5. Display-grade rendering conventions

Rendering constants feed **display only**, never dynamics:

- **Central-body radii.** Earth 6378137.0 m (WGS 84 semi-major axis, NGA
  TR8350.2); Moon 1737400.0 m (IAU/IAG mean radius) and Mars 3396190.0 m
  (IAU/IAG equatorial radius), both per Archinal et al. 2018 (Celest. Mech.
  Dyn. Astron. 130:22). HUD altitude is geocentric (|r| - radius), not
  geodetic.
- **Earth rotation.** The groundtrack and coastline co-rotate by the Earth
  Rotation Angle only: ERA(Tu) = 2 pi (0.7790572732640 +
  1.00273781191135448 Tu), Tu = JD(UT1) - 2451545.0 (IERS Conventions 2010,
  eq. 5.15), with UTC taken as UT1 and precession-nutation/polar motion
  neglected -- about a degree of longitude at 2020s epochs. Bodies without a
  rotation model show the inertial sub-point (stated in the help overlay).
- **Attitude.** `q_i2b` is the project frame transformation (rotation
  kernel: v_b = q^-1 v q). three.js applies quaternions as the active
  sandwich q v q^-1, which under this convention is the body-to-inertial
  map, so logged components are used directly.
- **Force arrows.** Body-frame forces rotate into the scene through the
  interpolated attitude; arrow lengths are log-scaled across the run's
  magnitude range (exact magnitudes shown in the legend); values step-hold
  at the last logged record. Lighting is fixed, not the run's sun geometry.

## 6. Viewer features (FR-19 surface)

- **Scrub bar** with one tick per log event (hover shows detail, time, and
  code); click/drag to scrub.
- **Play/pause** (button or space), **speed** 0.1x-1000x on a logarithmic
  slider.
- **Four camera modes**, named in the help overlay: *free orbit* (drag +
  wheel about the central body), *chase* (behind the velocity vector),
  *body-fixed* (camera rides the logged body frame), *nadir* (above the
  spacecraft on the body-center line). Keys 1-4.
- **Toggleable overlays**: axes triads (spacecraft body + inertial),
  velocity vector, thrust/per-source force vectors (disabled when the log
  has no `forces` group), trail, groundtrack. Keys a/v/f/t/g.
- **HUD**: UTC epoch (section 2), mission elapsed time, geocentric
  altitude, speed, current event window, and the standing
  non-physical-interpolation caveat.

## 7. CLI

```
star view <run.srlog> [-o out.html]
```

Default output is the input path with an `.html` suffix. The output is
overwritten if present (a derived artifact, regenerable bit-identically).
Exit codes follow the CLI contract: 0 success, 1 runtime errors (missing or
corrupt log). Stdout reports keyframes kept/total, the bound, the measured
maximum error, and the output path/size.
